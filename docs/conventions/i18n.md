# i18n Conventions

## Quick Reference

```vue
<template>
  <p>{{ $t('common.loading') }}</p>
  <p>{{ $t('cart.add_to_cart') }}</p>
  <p>{{ $t('elements.showing_n_of_total', { n: 16, total: 96 }) }}</p>
  <input :placeholder="$t('nav.search_products')" />
</template>

<script setup>
const { t } = useI18n();
const message = t('auth.login_failed');
</script>
```

## File Structure

```
app/locales/
  en.json    # English (default)
  sv.json    # Swedish
```

Locale files are **lazy-loaded** — only the active locale is included in the bundle.

## Key Naming

| Rule                        | Example                        |
| --------------------------- | ------------------------------ |
| snake_case                  | `add_to_cart`, not `addToCart` |
| Nested by domain            | `auth.login`, `cart.checkout`  |
| Descriptive, not positional | `login_failed`, not `error_1`  |

### Domains

| Domain     | Purpose                                                              |
| ---------- | -------------------------------------------------------------------- |
| `account`  | Account portal: profile, orders, settings                            |
| `auth`     | Authentication: login, logout, register                              |
| `cart`     | Shopping cart: add, remove, checkout                                 |
| `common`   | Shared UI: loading, error, cancel, save, search                      |
| `discount` | Discount labels, campaign badges, volume pricing, negotiated pricing |
| `elements` | Component showcase page                                              |
| `nav`      | Navigation: home, products, cart                                     |
| `product`  | Product pages: details, description, variants                        |

## Usage Patterns

### In Templates

```vue
<!-- Simple key -->
<p>{{ $t('common.loading') }}</p>

<!-- With interpolation -->
<p>{{ $t('elements.showing_n_of_total', { n: count, total: totalCount }) }}</p>

<!-- In attributes (use :attr binding) -->
<input :placeholder="$t('nav.search_products')" />
```

### In Script

```typescript
const { t } = useI18n();
const errorMessage = t('auth.login_failed');
```

### In Pinia Stores

Do **not** call `useI18n()` inside Pinia actions — it requires a Vue composition API context that may not be available in tests or SSR edge cases.

Instead, store the **translation key** as a string. The component that displays the error uses `$t()` to translate it:

```typescript
// Store: use key string as fallback
actions: {
  async login(creds) {
    try { /* ... */ } catch (err) {
      this.error = err instanceof Error ? err.message : 'auth.login_failed'
    }
  }
}
```

```vue
<!-- Component: translate when displaying -->
<p v-if="authStore.error">{{ $t(authStore.error) }}</p>
```

## Adding a New Locale

1. Create `app/locales/{code}.json` with all translation keys
2. Add the locale to `nuxt.config.ts`:
   ```typescript
   i18n: {
     locales: [
       // ...existing
       { code: 'de', name: 'Deutsch', file: 'de.json' },
     ],
   }
   ```
3. Ensure the tenant config includes the Geins locale code (e.g. `de-DE`) in its available locales

## Adding Translation Keys

1. Add the key to **all** locale files (`en.json`, `sv.json`, etc.)
2. Use the appropriate domain namespace
3. Keep values concise — these are UI labels, not documentation

## Tenant Locale Sync

- The `i18n-locale` plugin reads the tenant's `locale` field and calls `setLocale()`
- The `locale` cookie persists the user's preference across requests
- Server-side API routes can read the cookie via `getRequestLocale()` to pass the correct `languageId` to Geins GraphQL queries

## Server-Side Locale Override

All GraphQL services use `getRequestChannelVariables(sdk, event)` to automatically pipe the user's locale and market preferences into queries:

```typescript
// In a service function:
import { getTenantSDK, getRequestChannelVariables } from './_sdk';

export async function getProducts(options, event: H3Event) {
  const sdk = await getTenantSDK(event);
  return sdk.core.graphql.query({
    queryAsString: loadQuery('products.graphql'),
    variables: { ...options, ...getRequestChannelVariables(sdk, event) },
  });
}
```

This reads from:

- **Locale**: `locale` cookie (set by `@nuxtjs/i18n`)
- **Market**: `market` cookie (set via `setMarketCookie()`)

If no cookie is set, falls back to the tenant's default locale/market from SDK settings.

## CMS Content Language Behavior

CMS content in Geins has per-item language filters. The behavior differs by content type:

| Type                                         | Language behavior                                                                | Fallback                                                      |
| -------------------------------------------- | -------------------------------------------------------------------------------- | ------------------------------------------------------------- |
| **Widget areas** (start page, content areas) | Filtered by `languageId`. May only exist for one language.                       | Yes — retries without `languageId` so SDK uses default locale |
| **Menus**                                    | Filtered by `languageId`.                                                        | Yes — same fallback as widget areas                           |
| **Pages**                                    | Language-specific. Different aliases per language (e.g. `om-oss` vs `about-us`). | No — 404 if page doesn't exist for the requested locale       |

The fallback strips `languageId` from both the query variables **and** the `RequestContext` (the SDK merges both). Preview mode is preserved through the fallback chain.

CMS-generated page URLs use `/{locale}/{alias}` format (no market). Our `stripGeinsPrefix()` utility handles this alongside the standard `/{market}/{locale}/{path}` format.

## Alias Language Behavior

Geins's per-language alias queries (`product`, `categoryByAlias`, `brandByAlias`, `discountCampaignByAlias`) return `null` when the entity is not published in the requested language channel. Translating the name and texts in the admin is not sufficient on its own; the entity must also be activated for that channel.

| Scenario                                               | Behavior                                                                                                 |
| ------------------------------------------------------ | -------------------------------------------------------------------------------------------------------- |
| Translation exists for requested locale                | Returns translated fields                                                                                |
| Requested locale returns `null`, default locale has it | Service retries once with the SDK default `languageId`. PDP / PLP renders with default-language content. |
| Both locales return `null`                             | Throws `NOT_FOUND`; route sets `Cache-Control: no-store` so the CDN does not pin the miss                |

The retry lives in `server/services/_locale-fallback.ts` (`resolveWithLocaleFallback`). Every alias-resolving service composes it; do not reimplement the retry inline. Add a new caller by passing `{queryPath, variables: {alias}, serviceName}`. Route handlers and components should treat a non-null entity as authoritative regardless of which locale it came from.

See [ADR-005, Alias Language Fallback](../adr/005-internationalization.md#alias-language-fallback) for the full rationale.

## Switching Language on Entity Pages (Products, Categories, Brands)

Entity URL slugs in Geins are language-specific: product 1335 is `skarkant` in
Swedish and `cutting-edge` in English; category `kategori-1` in Swedish is
`category-1` in English. A path-preserving switch (carry the current slug under
the new `/{market}/{locale}/` prefix) therefore lands on a slug that does not
exist in the target language. The server-side alias fallback then renders the
**default-language** content under the new locale prefix, which looks like the
switch silently did nothing. To switch correctly the storefront must resolve the
real target-language URL.

### `alternativeUrls`

Geins exposes per-language URLs on the same query the page already runs (no
extra round-trip). Product and `PageInfo` (category/brand) types carry an
`alternativeUrls { channelId country culture language url }` field
(GraphQL `GeinsAlternativeUrlTypeType`). Each entry is one language/market/channel
variant of the entity's URL.

### `useLocaleAlternates`

`app/composables/useLocaleAlternates.ts` decouples where the per-locale URLs come
from (the entity query) from how the switcher consumes them. It is
`useState`-backed (SSR-safe, shared per request) and **clears on every client
route change**, so a previous page's entity URLs can never leak into the switcher.

- **Publisher contract**: the page calls
  `setAlternates(entries, { type })` where `type` is `'product' | 'category' | 'brand'`.
  The entity type is required because it determines which route type prefix to
  inject (see step 3 below). Publishers are the PDP and PLP components.
- **Consumer contract**: `hrefFor(short)` returns the published routable path for
  a short locale code, or `undefined` when none was published.

`setAlternates` runs a three-step mapping over the entries:

1. **Market filter.** Geins returns entries spanning many markets and channels
   (`se`, `fi`, `dk`, `us`, `gb`, ...). Keep only entries whose URL first path
   segment equals the active market, otherwise a last-wins map can land the user
   on the wrong market (e.g. `/fi/en/...`).
2. **Culture/language to short code.** Map the BCP-47 `culture` (e.g. `en-US`) to
   the short URL locale code (`en`), falling back to `language` when `culture` is
   missing or does not collapse to a two-letter code. Keep the code only when it
   is in the tenant's available locales. Short codes live in URLs; BCP-47 stays at
   the GraphQL boundary.
3. **Routable-path normalization.** Geins "pretty" SEO paths often **omit** the
   `/p/` `/c/` `/b/` type prefix (e.g.
   `/se/en/materials/branch-pipes/manifold-150-150-88`), which would 404 in the
   app router. Inject the entity's type prefix after `/{market}/{locale}/` when it
   is absent, and leave it untouched when the path already carries it (some
   tenants include it: `/se/en/p/category-1/cutting-edge`). This is why the
   publisher must pass the entity type.

### `LocaleSwitcher` consumption

`app/components/shared/LocaleSwitcher.vue` builds each `<a href>` from
`localeHref(loc)`:

- When `hrefFor(loc)` returns an alternate, the switch lands on the real
  target-language slug.
- Otherwise it falls back to the existing clean-path build
  `/${market}/${loc}${getCleanPath()}`. Non-entity pages (home, cart, CMS,
  account) never publish alternates, so they are unchanged.

### Last-resort server fallback

When an entity is genuinely untranslated, or a user manually types a
`/se/en/<sv-slug>` URL, no alternate exists. The server-side default-locale
fallback in `server/services/_locale-fallback.ts` still applies as the graceful
last resort, rendering default-language content rather than a 404. There is no
301 redirect.

See [ADR-005, Entity-Page Locale Switching via alternativeUrls](../adr/005-internationalization.md#entity-page-locale-switching-via-alternativeurls)
for the decision and consequences.
